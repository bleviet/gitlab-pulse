### 1. The Concept: "The AI Sidecar"

We treat the AI conversation not as temporary session data, but as **persistent documentation**. A summary generated by the AI becomes a tangible artifact linked to the issue, just like a label or a comment.

**The Workflow:**

1. **Select:** User clicks an issue in the dashboard (Flow > Issue Drill-down).
2. **Check:** System looks up `data/ai/chat_{issue_id}.parquet`.
   * *Case A (New):* No entry exists. -> Show "✨ Generate".
   * *Case B (Stale):* Entry exists, but `issue.updated_at > conversation.ref_timestamp`. -> Show "⚠️ Content changed".
   * *Case C (Fresh):* Entry exists and is current. -> Show summary and chat.
3. **Interact:** User chats with Ollama, context-aware of the issue.
4. **Persist:** Every message is saved back to Parquet.

#### 2.2. Schema Definition (`app/ai/models.py`)

We need a Pydantic model to structure the conversation storage.

```python
from pydantic import BaseModel, Field
from datetime import datetime
from typing import List, Optional

class ChatMessage(BaseModel):
    role: str = Field(..., description="Role of the sender: 'user' or 'assistant'")
    content: str
    timestamp: datetime = Field(default_factory=datetime.utcnow)

class IssueConversation(BaseModel):
    issue_id: int
    project_id: int

    # Staleness Logic
    ref_issue_updated_at: datetime  # The updated_at of the issue when this summary was generated
    generated_at: datetime = Field(default_factory=datetime.utcnow)

    # Content
    summary_short: str
    chat_history: List[ChatMessage] = Field(default_factory=list)
```

### 3. Technical Specification (TSD) for Layer 4

#### Component 1: The AI Service (`app/ai/service.py`)
This service wraps the connection to your local Ollama instance.

* **Endpoint:** `http://localhost:11434`
* **Health Check:** `GET /` (returns 200 OK if running).
* **Model List:** `GET /api/tags` (returns installed models).

**Key Methods:**
1. `check_health() -> bool`: Returns False if Ollama is down.
2. `get_available_models() -> List[str]`: Returns list of model names (e.g. `['llama3:latest', 'mistral:latest']`).
3. `chat(prompt, context, model) -> str`: Sends request to specific model.
* **Prompt Engineering:**

#### Component 2: The UI Pattern (Master-Detail) in Streamlit

Since you cannot easily put a "Chat Bubble" *inside* a `st.dataframe` cell that triggers a popup, the standard pattern in Streamlit is **Selection + Sidebar/Expander**.

**Implementation Logic in Layer 3:**

```python
# app/dashboard/views/drilldown.py

# 1. The Table (Master)
selection = st.dataframe(
    df,
    selection_mode="single-row",
    on_select="rerun",
    key="issue_drilldown_table"
)

# 2. The Detail View (Slave)
if selection and selection.rows:
    selected_idx = selection.rows[0]
    selected_row = df.iloc[selected_idx]
    issue_id = int(selected_row["id"])
    issue_iid = selected_row["iid"]

    with st.sidebar:
        st.header(f"🤖 AI Assistant: Issue #{issue_iid}")

        # --- 1. Connection Check ---
        if not ai_service.check_health():
            st.error("🔴 Ollama is offline. Ensure it is running on port 11434.")
            st.stop()  # Stop rendering the rest of the sidebar
        
        # --- 2. Model Selection ---
        available_models = ai_service.get_available_models()
        if not available_models:
            st.warning("⚠️ No models found. Run `ollama pull llama3`.")
            st.stop()

        selected_model = st.selectbox(
            "Select Model", 
            available_models, 
            index=0
        )

        # Load AI Data
        conversation = ai_service.get_conversation(issue_id)

        # Staleness Check
        is_stale = False
        if conversation:
            issue_updated = pd.to_datetime(selected_row["updated_at"], utc=True)
            ref_updated = pd.to_datetime(conversation.ref_issue_updated_at, utc=True)
            if issue_updated > ref_updated:
                st.warning("⚠️ Issue updated since last summary.")
                is_stale = True

        # Actions: Generate or Regenerate
        if not conversation or is_stale:
            btn_label = "🔄 Update Summary" if is_stale else "✨ Generate Summary"
            if st.button(btn_label, type="primary"):
                with st.spinner(f"Analyzing with {selected_model}..."):
                    try:
                        ai_service.generate_summary(selected_row, model=selected_model)
                        st.success("Summary generated!")
                        st.rerun()
                    except Exception as e:
                        st.error(f"AI Service Error: {e}")

        # Chat Interface
        if conversation:
            with st.expander("📝 Summary", expanded=True):
                st.markdown(conversation.summary_short)

            st.divider()
            st.markdown("### Chat History")

            for msg in conversation.chat_history:
                with st.chat_message(msg.role):
                    st.markdown(msg.content)

            if prompt := st.chat_input("Ask about this issue..."):
                with st.chat_message("user"):
                    st.markdown(prompt)

                with st.spinner("Thinking..."):
                    response = ai_service.chat(prompt, context=conversation, model=selected_model)
                    st.rerun()
```

### 4. Architecture Decision Records (ADR)

#### 4.1. Why Local Ollama?

* **Privacy:** Firmware logs and bug descriptions often contain sensitive IP or unreleased feature details. Sending this to OpenAI/Cloud APIs is a security risk. A local server ensures data never leaves the lab network.
* **Cost:** Running Llama 3 locally is free, enabling unlimited summarization without budget concerns.

#### 4.2. Why Parquet for Chat History?

* **Consistency:** The entire platform uses Parquet ("The Lakehouse Pattern"). Introducing SQLite just for chat would complicate the tech stack (e.g., needing ORMs like SQLAlchemy).
* **Performance:** We load the conversation history by `issue_id`. Parquet's columnar compression works well for text.

#### 4.3. Staleness Strategy (Timestamp vs. Hash)

* **Decision:** We use `updated_at` **Timestamp comparison**.
* **Why:** Hashing the description requires loading the large description text into memory every time to compute the hash. Comparing timestamps is an O(1) integer comparison, which fits our high-performance requirement.
